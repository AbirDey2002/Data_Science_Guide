Feature Engineering

Study guide from a guided learning session — March 2026

Machine Learning Statistics Linear Algebra Deep Learning

Why Feature Engineering

The process of using domain knowledge and data manipulation to create, transform, or select input variables that make ML models perform better. Raw data is almost never model-ready.

When to Use

Before any modelling When raw data has skew, mixed types, or missingness When model performance plateaus

Key Principles

  • Models don't understand context — you have to encode it (e.g. extract is_weekend from a timestamp)
  • Feature engineering often outperforms algorithm selection in impact
  • Better features reduce model complexity and data requirements
  • It is a feedback loop — the first model tells you where features are failing
Garbage in, garbage out. No algorithm compensates for poorly represented data.

Where Feature Engineering Fits in the Pipeline

Feature engineering picks up where EDA ends — it is the translation layer between data understanding and model consumption.

Three Zones

ZoneWhat Happens
Pre-modellingTransformations, encoding, scaling — making data digestible
Model-awareFeatures crafted knowing the model's assumptions (linear vs tree)
Post-first-modelUsing residuals, importances, errors to engineer new features iteratively
Feature engineering is not a one-time step — it is a feedback loop driven by model outputs.

Tree-based models are indifferent to skew. Linear models, KNN, and SVMs care a lot. Always ask: does this model care about distribution?

Interaction & Polynomial Features

Explicitly encoding that two variables together carry a signal neither holds alone. Most valuable in linear models — tree models discover interactions implicitly. Best interaction features come from domain knowledge, not brute-force expansion.

Temporal Features

Raw datetimes are useless. Extract: hour of day, is_weekend, lag features, rolling means. Use sin/cos encoding for cyclical features so that 11pm and 1am are numerically close.

Feature Selection

Knowing what to remove is as important as knowing what to create. Three families of methods:

Filter Methods — Statistics First, Model Never

TestUse Case
Pearson correlationContinuous feature vs continuous target
Spearman correlationNon-linear monotonic or ordinal data
ANOVA F-testContinuous feature vs categorical target
Chi-squaredCategorical feature vs categorical target
Mutual InformationAny type — captures non-linear dependencies
Filter methods evaluate features in isolation — they miss interaction effects entirely. Use for fast pruning only.

Wrapper Methods — Let the Model Decide

RFE (Recursive Feature Elimination), forward/backward selection. Uses model performance as the selection criterion. Expensive but captures interactions. Risk: overfitting the feature selection if not wrapped inside proper CV.

Embedded Methods — Selection During Training

Lasso (L1) drives irrelevant coefficients to zero. Ridge (L2) shrinks but doesn't zero out. Tree-based importances (Gini or permutation). Elastic Net combines L1 + L2. Most practical in real workflows — selection and modelling in one pass.

Practical Order of Operations

EDA findings → Filter methods to prune noise → Embedded methods in first model → Wrapper methods on shortlist if squeezing further

Gini-based importance is biased toward high-cardinality features. Correlated features split importance between themselves. Permutation importance is far more reliable.

Reading and Interpreting Findings

Feature Importance Flavours

MethodQuestion AnsweredLimitation
Gini / Split-basedHow often did the model use this feature?Biased toward high cardinality
Permutation importanceHow much does performance drop when shuffled?Correlated features dilute each other
SHAP valuesHow much did this feature affect this specific prediction?Computationally heavier

SHAP is the gold standard — it gives both global and per-prediction explanations, grounded in Shapley values from game theory.

Partial Dependence Plots (PDPs) and ICE

PDPs show the shape of the relationship between a feature and the target (averaged across all observations). ICE plots show the same per individual row — if ICE lines diverge wildly, interaction effects are present that PDP is hiding.

Ablation Testing

Train with and without a feature, compare CV scores. If the delta is within your confidence interval, the feature is not contributing anything real.

Red Flags

  • Suspiciously high importance — check for target leakage
  • Two features with split importance — likely correlated, check VIF
  • Feature important in train but not validation — overfitting, possibly high-cardinality ID column

Interpretability Stack

SHAP summary plot → SHAP dependence plot → ICE plots → Ablation tests → VIF / correlation matrix

PCA — Mathematical Internals

PCA finds directions in feature space along which data varies most, then projects data onto the top k such directions.

Step-by-Step

  1. Center the data: subtract column means from X
  2. Compute covariance matrix: $C = \frac{1}{n} X_c^T X_c$
  3. Eigen decomposition: solve $C \cdot v = \lambda \cdot v$ for all eigenpairs
  4. Sort and select: rank eigenvectors by eigenvalue descending, keep top k
  5. Project: $X_{reduced} = X_c \cdot W$ where W contains the top k eigenvectors
$$C = \frac{1}{n} X_c^T X_c \qquad C \cdot v = \lambda v \qquad X_{red} = X_c \cdot W$$

C = covariance matrix (m×m) · v = eigenvector · λ = eigenvalue · W = matrix of top k eigenvectors

Geometric Intuition

The covariance matrix encodes in which directions data spreads. Eigen decomposition finds those exact directions and how much spread lies in each. PCA rotates your coordinate system to align with the natural axes of data variance, then discards the least informative axes.

Scaling is not optional — features on different scales will dominate the covariance matrix. Always StandardScale before PCA.

Covariance matrix: $C = \begin{pmatrix} 8 & 48 \\ 48 & 336 \end{pmatrix}$

Strong off-diagonal (48) signals high redundancy — the two features are largely saying the same thing.

Eigen decomposition yields:

  • $\lambda_1 = 342.9$, $v_1 = [0.141, 0.990]$ — PC1: overall performance axis
  • $\lambda_2 = 1.1$, $v_2 = [0.990, -0.141]$ — PC2: residual noise
$$\text{PC1 explains} = \frac{342.9}{342.9 + 1.1} = 99.7\%$$

One component captures nearly all information. The 5×2 matrix reduces to a 5×1 vector with 99.7% variance retained.

Deriving Eigenpairs from C

The eigen equation $C \cdot v = \lambda v$ rearranges to $(C - \lambda I) \cdot v = 0$. For a non-trivial solution, the matrix must be singular:

$$\det(C - \lambda I) = 0$$

Setting the determinant to zero gives the characteristic polynomial. Its roots are the eigenvalues.

For our 2×2 C matrix

Expanding the determinant:

$$(8 - \lambda)(336 - \lambda) - (48)(48) = 0 \implies \lambda^2 - 344\lambda + 384 = 0$$ $$\lambda = \frac{344 \pm \sqrt{344^2 - 4 \cdot 384}}{2} = \frac{344 \pm 341.76}{2}$$

Gives λ₁ = 342.9 and λ₂ = 1.1

Finding the Eigenvectors

Substitute each λ back into $(C - \lambda I) \cdot v = 0$ and solve for v. The resulting vector is proportional to the eigenvector — normalize it to unit length by dividing by its magnitude $\|v\| = \sqrt{v_1^2 + v_2^2}$.

Eigenvectors have arbitrary scale — only direction matters. Normalizing to unit length ensures projections are geometrically clean. Use np.linalg.eigh (not eig) for symmetric covariance matrices — it is faster and numerically more stable.

How Many Components to Keep

Method 1 — Variance Threshold (95% Rule)

Keep enough components to explain 95% (or 90%) of total variance. Quick and widely used.

A component explaining only 3% of variance might carry exactly the signal your model needs. This rule answers the wrong question for downstream tasks.

Method 2 — Scree Plot and Elbow

Plot eigenvalues in descending order. Components before the elbow carry signal; after it, noise. Subjective — elbows are often unclear in real data.

Method 3 — Kaiser Criterion

Keep only components with eigenvalue > 1. Logic: on standardized data, each original feature contributes eigenvalue of 1. A component below this threshold explains less than a single original feature. Conservative — use as a lower bound, not a final answer.

Method 4 — Cross-Validation (Most Rigorous)

Wrap PCA in a Pipeline, grid-search over candidate values of k, use model CV performance as the criterion. This answers the correct question: how many components make the downstream model perform best?

Decision Framework

Scree plot + 95% threshold to get a range → Kaiser as a floor → CV over that range → sanity check that reducing further does not significantly hurt performance.

PCA must always live inside a Pipeline. Fitting PCA on the full dataset before train/test split is data leakage.

Autoencoders — Non-Linear PCA

A neural network that compresses data to a low-dimensional bottleneck Z and then reconstructs the original input. The bottleneck is the learned dimensionality-reduced representation.

Architecture

Input X → [Encoder] → Bottleneck Z → [Decoder] → Reconstructed X̂

$$\mathcal{L} = \frac{1}{n} \sum_{i=1}^n \|X_i - \hat{X}_i\|^2$$

MSE reconstruction loss — network minimizes this, forcing Z to retain maximum useful information

Why It Beats PCA for Non-Linear Data

PCA can only find straight-line directions (linear combinations). An autoencoder can learn curved manifolds — finding that data lives on a complex surface in high-dimensional space and unrolling it cleanly.

PCA vs Autoencoder

PCAAutoencoder
RelationshipLinear onlyNon-linear
InterpretabilityComponents inspectableBottleneck is a black box
Data requirementWorks on small datasetsNeeds sufficient training data
SpeedVery fastSlower — requires training
Anomaly detectionPoorExcellent — high reconstruction error = anomaly

Notable Variants

  • VAE (Variational): encodes to a distribution (mean + variance), not a point — smooth latent space, useful for generative modelling
  • Sparse Autoencoder: penalizes non-zero bottleneck activations — encourages distinct, interpretable features
  • Denoising Autoencoder: corrupts input with noise then trains to reconstruct clean version — forces more robust feature learning
For tabular ML, PCA is usually the right call. Autoencoders shine when data has genuine non-linear structure — images, audio, sensor signals, text embeddings.

Formula Reference Sheet

Covariance Matrix

$$C = \frac{1}{n} X_c^T X_c$$

Variables: Xc = mean-centered data matrix (n×m), C = m×m covariance matrix

Used when: Step 2 of PCA — captures how every feature co-varies with every other

Eigen Equation

$$C \cdot v = \lambda \cdot v \quad \Leftrightarrow \quad \det(C - \lambda I) = 0$$

Variables: v = eigenvector (direction of variance), λ = eigenvalue (amount of variance)

Used when: Finding principal components from the covariance matrix

PCA Projection

$$X_{reduced} = X_c \cdot W, \quad W \in \mathbb{R}^{m \times k}$$

Variables: W = matrix of top k eigenvectors, k = chosen number of components

Used when: Step 5 of PCA — projecting data onto new coordinate axes

Explained Variance Ratio

$$\text{EVR}_k = \frac{\sum_{i=1}^k \lambda_i}{\sum_{j=1}^m \lambda_j}$$

Variables: λᵢ = ith eigenvalue sorted descending, m = total features, k = selected components

Used when: Deciding how many PCA components to retain

Characteristic Polynomial (2×2)

$$(a - \lambda)(d - \lambda) - bc = 0 \implies \lambda^2 - (a+d)\lambda + (ad - bc) = 0$$

Variables: a, d = diagonal entries, b, c = off-diagonal entries of C

Used when: Manually deriving eigenvalues from a 2×2 covariance matrix

Autoencoder Reconstruction Loss

$$\mathcal{L} = \frac{1}{n} \sum_{i=1}^n \| X_i - \hat{X}_i \|^2$$

Variables: X = original input, X̂ = decoder output, n = batch size

Used when: Training an autoencoder — minimizing this forces the bottleneck to retain useful structure

L1 Regularization (Lasso)

$$\mathcal{L}_{lasso} = \text{MSE} + \alpha \sum_{j=1}^m |\beta_j|$$

Variables: βj = feature coefficient, α = regularization strength

Used when: Embedded feature selection — drives irrelevant coefficients to exactly zero

Code Reference

sklearn PCA inside a Pipeline (production-safe)

from sklearn.preprocessing import StandardScaler
from sklearn.decomposition import PCA
from sklearn.pipeline import Pipeline
import numpy as np

X = np.random.randn(200, 10)

pipe = Pipeline([
    ('scaler', StandardScaler()),       # MUST scale before PCA
    ('pca', PCA(n_components=0.95))     # retain 95% of variance
])
X_reduced = pipe.fit_transform(X)

print(pipe['pca'].n_components_)
print(pipe['pca'].explained_variance_ratio_.cumsum())

PCA from scratch (maps directly to math)

import numpy as np

def pca_from_scratch(X, k):
    # Step 1: center → X_c = X - μ
    X_c = X - X.mean(axis=0)

    # Step 2: covariance matrix → C = (1/n) Xᵀ X
    C = (X_c.T @ X_c) / len(X_c)

    # Step 3: eigen decomposition → C·v = λ·v
    eigenvalues, eigenvectors = np.linalg.eigh(C)

    # Step 4: sort descending, pick top k
    idx = np.argsort(eigenvalues)[::-1]
    W = eigenvectors[:, idx[:k]]          # W ∈ ℝ^(m×k)

    # Step 5: project → X_reduced = X_c · W
    return X_c @ W

Cross-validation to select number of PCA components

from sklearn.pipeline import Pipeline

Autoencoder in PyTorch

import torch
import torch.nn as nn

class Autoencoder(nn.Module):
    def __init__(self, input_dim, bottleneck_dim):
        super().__init__()

        self.encoder = nn.Sequential(
            nn.Linear(input_dim, 128),
            nn.ReLU(),
            nn.Linear(128, 64),
            nn.ReLU(),
            nn.Linear(64, bottleneck_dim)   # → Z (compressed)
        )

        self.decoder = nn.Sequential(
            nn.Linear(bottleneck_dim, 64),
            nn.ReLU(),
            nn.Linear(64, 128),
            nn.ReLU(),
            nn.Linear(128, input_dim)        # → X̂ (reconstructed)
        )

    def forward(self, x):
        z = self.encoder(x)
        x_hat = self.decoder(z)
        return x_hat, z

model = Autoencoder(input_dim=784, bottleneck_dim=2)
criterion = nn.MSELoss()
optimizer = torch.optim.Adam(model.parameters(), lr=1e-3)

for epoch in range(50):
    x_hat, z = model(X_train)
    loss = criterion(x_hat, X_train)
    optimizer.zero_grad()
    loss.backward()
    optimizer.step()

_, Z = model(X_test)   # Z = compressed representation
